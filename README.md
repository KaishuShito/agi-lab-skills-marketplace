## 世界のトップOSSが見逃すバグを確実に捉える
テストもCIもレビューも通るのにいつの間にか潜伏するバグ。世界トップのメンテナたちが見逃したバグも複数検知して公式にバグとして認められ、修正が取り込まれる。

これまでのバグ検出ツールと違った「**構造矛盾(情報損失)**」という概念を使って自動検出します。

## 使い方

```
delta scan                    # 直近3ヶ月の変更ファイルを自動スキャン
delta scan --since 1year      # 直近1年分の変更をスキャン
delta scan --scope pr         # PR全体の変更をスキャン（ベースブランチから自動検出）
delta scan --scope smart      # git履歴から高リスクファイルを優先スキャン
delta scan --scope wide       # 全ファイルをバッチ分割スキャン
delta scan --autofix          # 検出 + 修正コード自動生成
delta view                    # ダッシュボードを表示
```

`delta scan` と打つだけ。何を探すかも、どこを見るかも、エージェントが自分で判断します。
「PRレビューして」と言えば自動で `--scope pr` が選択されます。

### 自律実行の流れ

```
delta scan（Enter 1回）
  → 直近3ヶ月の git history から変更ファイル特定（--since で期間変更可）
  → import 追跡で 1-hop 依存を収集
  → 6パターンの構造矛盾を LLM で検出
  → 負債スコア算出
  → ダッシュボード生成（バッチ完了ごとにリアルタイム更新）

delta scan --scope pr
  → merge-base 算出でPR全体の差分を取得
  → バッチ分割 → プログレスバー付き逐次実行
  → 各バッチ完了時にダッシュボード + findings 自動反映
```

### 検出 → 修正 → PR（フル自律）

```
delta scan --autofix          # 検出 + 自動修正
debt-loop --ids F001,F002     # 選んだ finding を修正 → branch → PR 作成
```

## 6つの検出パターン

| # | パターン | 例 | なぜ起きる |
|---|---------|---|-----------|
| ① | **非対称デフォルト** | 会員登録で名前を空欄にするとプロフィール画面に「null」と表示される | 保存と表示で「空」の扱いが違う<br>*← 入力パスと出力パスでデフォルト値が非対称* |
| ② | **意味的不一致** | 注文ステータス「0」が画面Aでは「未処理」、画面Bでは「キャンセル済」になる | 同じ名前が場所によって別の意味<br>*← 共有名の意味がモジュール間で暗黙に分岐* |
| ③ | **仕様乖離** | 「全APIに認証必須」と書いてあるが認証なしで叩けるAPIがある | ドキュメントと実装がズレている<br>*← 仕様と実装の同期が手動依存で維持されない* |
| ④ | **ガード欠落** | 新規投稿にはXSS対策があるが編集画面にはない | 片方のパスだけチェックが抜けている<br>*← 並行パスへのガード伝搬が保証されない* |
| ⑤ | **設定干渉** | セッション有効期限30分 × 自動保存間隔40分で下書きが毎回消える | 独立に見える設定が裏で矛盾する<br>*← 設定間の暗黙の制約が文書化されていない* |
| ⑥ | **順序依存** | ログイン直後だけ「あなたへのおすすめ」が他人のデータで表示される | 特定の条件で実行順が入れ替わる<br>*← 実行順序の前提がコードに明示されていない* |

共通する原因：**開発者はスコープを絞って作業する**。機能Aを修正してテストが通れば完了。しかし機能Aと暗黙の前提を共有する機能Bが壊れていないかは、誰もチェックできていない可能性がある。DeltaLint はこの「スコープ外」を狙って検出する技術。

## なぜこれがこんなに機能するのか(背景技術)

この技術は、数学的に証明した構造崩壊理論をベースに作っています。(私の理論、失礼)<br>
[Structural Collapse as Information Loss: The Exponential Decay Mechanism under Accumulating Constraints](https://zenodo.org/records/18943286)（[PDF](Information_loss.pdf)） <br>
※東大松尾研OB(学際発表経験有)の方に査読してもらいました<br>

特定の条件を満たす構造は、構造矛盾によって急激に崩壊するという理論です。<br>
ソフトウェア本体を構造物と捉えて、矛盾があったときにバグとして炙り出てくるという理論応用になります。

LLM 11モデル・5ベンダーでの実験と、SAT問題での数学的検証により、構造矛盾が蓄積するとシステムの健全性は**指数関数的に**崩壊する（足し算ではなく掛け算）ことを確認しています。

DeltaLint はこの理論を使って、コードの中から「設計図の食い違い」を自動検出します。だから普通のリンターやテストでは見つからないバグが見つかります。

## 目的達成のための自律性

`delta scan` と打つだけで動く。その裏で、人間が設定しなくても正しく動くための判断を自動で行っている。

### どこをスキャンするか、自分で決める

| やっていること | 具体例 | 設計意図 |
|--------------|--------|---------|
| リポの規模に合わせて時間窓を調整 | 3ヶ月で30件以上 → 3ヶ月。10件未満 → 9ヶ月〜1年に自動拡張 | 小さいリポに「3ヶ月」は短すぎ、大きいリポに「1年」は遅すぎる |
| 変更頻度 × 参照数 × 最終更新日で優先順位をつける | 毎週変わる `auth.ts`（10ファイルから参照）→ 最優先 | よく変わり、多くから使われるファイルほどバグが入りやすい |
| ディレクトリを横断して均等にサンプリング | `api/` から3件、`db/` から3件、`ui/` から3件 | アルファベット順だと後半のディレクトリが永遠に後回しになる |
| 2回目以降は前回の結果を踏まえて戦略を変える | 初回: 広く探索。2回目: ホットスポット再検査 + 未踏領域 | 毎回同じでは収束しない。前回だけ見ると視野が狭まる |
| 新しい発見がなくなったら自動停止 | 直近20回で新規ゼロ → 「もう十分」と判断 | 終了タイミングを人間が決めなくていい |

### 何と何を比較するか、自分で決める

| やっていること | 具体例 | 設計意図 |
|--------------|--------|---------|
| 依存の遠さに応じて信頼度を下げる | 同じフォルダ内 → 95%。3ホップ先 → 61% | 遠い依存まで同じ重みだと無関係なファイル同士を比較して誤検出が増える |
| import文に書かれない依存を見つける | Springの `@Autowired` でDI注入されるサービス、Djangoのsignal、Railsの `before_action` コールバック等（8フレームワーク対応） | フレームワークはDI・イベント・フックで結合する。import解析だけでは見えない |
| 一緒に変更されるファイルのパターンを学習する | `handler.ts` と `validator.ts` が過去20回中18回一緒に変更 → 兄弟認定 | git履歴が教える「隠れた結合」はコード構造からは見えない |
| LLMに渡す文脈量を自動調整 | 直接依存だけなら80K文字。間接依存も辿るなら160Kまで自動拡張 | 少なすぎると見逃す。多すぎるとLLMの精度が落ちる |

### 壊れても結果を返す

| 壊れるもの | よくある状況 | 対処 |
|-----------|------------|------|
| LLMのAPI | タイムアウト、レート制限 | 自動リトライ + CLI→SDK→HTTPの3層フォールバック |
| LLMの出力 | JSONのはずがmarkdownで返ってくる | 4段階のパーサーが順に試行して部分的にでも結果を抽出 |
| gitの履歴 | CIのshallow clone（履歴1件） | ファイルサイズから変更頻度を推定 |
| 記録ファイル | JSONLの途中の行が壊れている | 壊れた行をスキップして残りを読む |
| 巨大リポジトリ | 36,000ファイルで終わらない | 10件ごとに途中保存。制限時間で中間結果を返して終了 |

### 同じバグを二度報告しない

| 重複の種類 | 例 | 判定方法 |
|-----------|---|---------|
| 完全一致 | 同じファイル・同じバグ | ファイル名+パターン+タイトルからハッシュID生成 |
| 言い換え | 「未使用のデフォルト値」≈「使われないデフォルト設定」 | テキスト類似度55%以上で同一判定 |
| 別角度から同じ実体 | パターン①で型不一致、パターン⑨でインターフェース不整合 → 同じ `twitter_profile` | コード中の関数名・ファイル名の重複率60%以上で統合 |

### 優先度を自分で決める

| 軸 | 何を測るか | 高い例 | 低い例 |
|----|----------|--------|--------|
| 負債係数 | バグの深刻さ | 高重要度 × 前提の非対称（①） | 低重要度 × デッドコード（⑦） |
| 解消価値 | 直したらどれだけ得か | 月5回変更・20ファイルから参照・修正が簡単 | 放置中・参照なし・大規模改修が必要 |
| 情報量 | この発見は新しい知見か | 初めてのパターン × 問題集中ファイル | 同パターンの10件目 × 孤立ファイル |
| 未発見推定 | まだ見つかっていないバグはあと何件か | 毎回新しいバグが出る（カバレッジ不足） | 既知の再検出ばかり（収束） |
